# 심야 병동 맵·멀티플레이 방 스펙

- 문서 버전: 1.2
- 기준일: 2026-07-23
- 상태: 서버 구현 기준
- 코드 기준점: `src/shared/map.ts`, `src/server/engine.ts`, `src/client/game/ThreeGameView.ts`

## 1. 맵 토폴로지

- 맵 전체 크기는 73×41 타일이다.
- 3타일 폭 세로 복도 5개와 가로 복도 3개가 교차해 작은 사각형 루프와 우회 경로를 만든다.
- 방은 복도 사각 셀 안에 2행×4열, 총 8개를 생성한다.
- 방의 문 방향은 상·하·좌·우 중 시드 기반으로 선택한다. 방 바닥과 복도 사이에는 문 한 칸만 통로로 남겨 벽 통과를 막는다.
- 플레이어·귀신 A\* 경로는 같은 `walkable` 타일 집합을 사용한다. 모든 방의 침대에서 리스폰 구역까지 문을 포함한 경로가 존재해야 맵을 승인한다.

## 2. 싱글 플레이 방

- 방 바닥은 최소 20칸, 최대 25칸이다.
- 4×5, 5×4, 5×5, 6×4와 모서리를 제거한 꺾인 형태를 섞는다.
- 방마다 침대 1개를 두고 나머지 19~24칸을 건설 가능 타일로 사용한다.
- 한 방에는 한 명만 머문다.

## 3. 멀티플레이 공유 방

- 모든 방은 정확히 25칸이다.
- 방마다 서로 떨어진 침대 2개를 두고 23칸을 건설 가능 타일로 사용한다.
- 한 방 최대 인원은 침대 수와 같은 2명이다. 생존자는 준비 시간 또는 전투 중 빈 침대 가까이에서 직접 점유하며, 서버가 시작 시점에 빈 침대를 자동 배정하지 않는다.
- 싱글전은 방 정원을 정확히 1명으로 제한한다. 따라서 봇 또는 플레이어가 이미 점유한 싱글 방의 다른 침대 좌표에는 들어갈 수 없다. 멀티플레이에서만 같은 방의 서로 다른 침대 두 개를 사용한다.
- 같은 방의 두 플레이어는 빈 타일에 각각 건설할 수 있고, 누가 설치했는지와 관계없이 포탑·문·발전기 등 설비를 강화할 수 있다.
- 각 플레이어는 자기 침대만 강화할 수 있다.
- 침대 골드는 해당 침대 점유자에게만 지급한다.
- 발전기 전력은 `BuildingState.ownerId`와 플레이어 ID가 같은 발전기에서만 지급한다.
- 다른 플레이어가 설비를 강화해도 설치 소유자는 바뀌지 않으므로 생산 귀속도 바뀌지 않는다.
- 같은 방 플레이어는 설비를 철거할 수 있다. 서버는 설치·강화 때마다 플레이어별 투자 골드와 전력을 기록하고, 철거 시 총 투자액의 70%를 각 투자자에게 분배한다.
- 철거된 `BuildingState`는 서버 스냅샷에서 제거되며 해당 타일은 즉시 다시 건설할 수 있는 빈 타일이 된다.

## 4. 대기실 방 수명주기와 권한

- 모든 플레이어는 대기실에서 `방 나가기`를 사용할 수 있다. 명시적 퇴장은 네트워크 단절과 다르게 처리하며 해당 플레이어의 재접속 토큰을 폐기한다.
- 방 나가기, 추방, 빈 방 종료 후에는 구형 플레이 방식 선택 화면을 거치지 않고 게임 홈으로 돌아간다.
- 싱글 플레이는 서버 연결용 내부 방 식별자를 사용하되 대기실에 `ROOM CODE`를 표시하거나 복사 기능을 제공하지 않는다. 초대 코드는 멀티플레이에서만 노출한다.
- 방장이 나가고 다른 실제 플레이어가 있으면 연결된 실제 플레이어 중 먼저 입장한 사람이 새 방장이 된다. 이전 방장은 자동 복귀하지 않으며 초대 코드를 다시 입력해야 참가할 수 있다.
- 마지막 실제 플레이어가 나가면 남아 있던 봇을 함께 제거하고 Durable Object 방 상태·알람을 삭제한다. 이후 사용자는 새 방을 만들 수 있다.
- 방장만 대기실의 봇을 제거하거나 다른 실제 플레이어를 추방할 수 있다. 추방된 플레이어의 재접속 토큰도 즉시 폐기한다.
- 봇 제거·추방·방장 승계 결과는 서버 스냅샷으로 모든 남은 참가자에게 동기화한다.

## 5. 귀신 목표 선택

- 방을 점유하지 못한 연결된 생존자가 한 명이라도 있으면 귀신은 문보다 해당 생존자를 우선 추적한다. 실제 플레이어를 봇보다 먼저 선택하고, 같은 유형 안에서는 귀신과 가까운 생존자를 선택한다.
- 이 복도 추적 중 귀신 이동속도는 기본·스테이지·변종·격노·감속 보정 뒤에 `1.35배`를 추가 적용한다. 모바일 10Hz 동기화에서도 이동이 끊겨 보이지 않도록 일반 생존자와 비슷한 속도로 유지한다. 접촉 공격은 한 번에 사망 처리한다.
- HP가 20% 이하가 된 귀신은 후퇴 중인 수호 포탑 집중 사격 피해를 1.45배만 받고, 기본 이동 보정의 1.30배로 현재 위치에서 가장 가까운 리스폰 패드까지 이동한다. 패드는 네 모퉁이와 상·하·좌·우 변 중앙에 하나씩, 총 8개다.
- 미점유 생존자가 없을 때만 귀신은 살아 있는 플레이어가 있는 방을 목표 후보로 사용한다.
- 목표가 필요할 때 후보 중 하나를 균등 무작위로 선택하며 첫 방이나 최근 방을 고정 우선하지 않는다.
- 쌍둥이 귀신은 후보 방이 2개 이상이면 다른 쌍둥이가 선택한 방을 제외하고 고른다.
- 목표 방의 문이 살아 있으면 문을 공격하고, 문이 파괴되면 그 방의 살아 있는 플레이어 중 가까운 플레이어를 공격한다.

## 6. 렌더링·조작

- 복도 판정은 단일 가로 띠가 아니라 `corridorTiles` 전체를 사용한다.
- 복도망 전체에 샘플 조명을 분산한다.
- 벽 높이는 본체 0.68, 상단 캡 0.73 유닛으로 제한한다.
- 방 안의 모든 침대는 독립 모델로 표시하고 선택 ID는 `bed:{roomId}:{bedIndex}`를 사용한다.
- 카메라는 점유 전 살아 있는 플레이어 추적, 점유 후 드래그·회전·0.5×~2.0× 확대/축소 규칙을 유지한다. 사망 뒤에는 즉시 추적을 해제해 관전 카메라를 드래그·회전·확대/축소할 수 있다.
- 설치·강화 패널 바깥을 누르면 패널과 선택 상태를 닫는다. 별도 취소 버튼은 두지 않는다.

## 7. 검증 조건

- 동일 시드는 동일 맵을 만든다.
- 방 8개와 리스폰 패드 8개가 생성되고, 모드별 침대 수가 일치한다.
- 모든 이동 가능 타일이 하나의 연결 성분이다.
- 각 방의 침대→플레이어 스폰 경로는 해당 방 문을 반드시 지난다.
- 멀티 두 명이 같은 방의 다른 침대에 배정되고, 공동 강화와 싱글 생산 귀속이 서버 테스트로 검증된다.
- 쌍둥이는 서로 다른 점유 방이 존재할 때 서로 다른 목표를 선택한다.
- 방장 퇴장 후 승계, 봇 제거, 플레이어 추방, 마지막 사람 퇴장 후 방 삭제가 실제 WebSocket 두 클라이언트 시나리오에서 검증된다.
- 설치·강화 투자액의 70% 환급과 같은 타일 재건설이 서버 테스트로 검증된다.
